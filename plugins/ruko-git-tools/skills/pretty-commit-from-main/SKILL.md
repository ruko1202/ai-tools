# Pretty commit from main

Превращает кашу из WIP-коммитов в аккуратную историю: схлопывает всё, что наработано
поверх `main`, обратно в рабочие изменения (`git reset --soft main`), а потом заново
раскладывает их на чистые коммиты, сгруппированные по смыслу. Ветка остаётся та же,
файлы не теряются — переписывается только история коммитов.

## Зачем это и почему так

`reset --soft` двигает только указатель ветки: индекс и рабочее дерево он не трогает.
Поэтому после `git reset --soft main` все изменения с момента ответвления от `main`
оказываются разом в staged — ровно тот холст, с которого удобно лепить коммиты заново.
Ни один файл при этом не пропадает. Цель упражнения — чтобы в итоге каждый коммит
рассказывал одну законченную мысль, а не «wip», «fix», «again».

## Перед стартом: предохранители

Операция переписывает историю, поэтому сначала убедись, что её вообще безопасно делать.

1. Это git-репозиторий: `git rev-parse --is-inside-work-tree`.
2. Текущая ветка — не `main`: `git rev-parse --abbrev-ref HEAD`. Ресетить `main` на саму
   себя бессмысленно; если мы на `main`, остановись и скажи об этом.
3. `main` существует: `git rev-parse --verify main`. Если в репозитории основная ветка
   называется иначе (`master`, `develop`), не угадывай молча — спроси, какую брать за базу.
4. **Главный предохранитель.** Проверь, что `main` — действительно предок текущей ветки:
   `git merge-base --is-ancestor main HEAD`.
   - Код возврата `0` → всё хорошо, `reset --soft main` соберёт только твои изменения.
   - Любой другой код → ветка отстала от `main` или разошлась с ней. Тогда reset на tip
     `main` затянет в staged чужие коммиты и/или откатит то, что уже влилось в `main`, —
     получится мусор. **Остановись**, объясни ситуацию и предложи сначала подтянуть ветку
     (`git rebase main` / `git merge main`), а уже потом запускать пересборку. Не делай
     reset вслепую.

## Шаг 1. Запомни точку отката

Перед любым reset сохрани, куда сейчас смотрит ветка, — это страховка, чтобы вернуть всё
одной командой, если разбивка пойдёт не туда:

```bash
OLD_HEAD=$(git rev-parse HEAD)
git log --oneline main..HEAD   # покажи пользователю, что именно схлопываем
```

## Шаг 2. Soft reset на main

```bash
git reset --soft main
```

После этого все наработки висят в staged, а история коммитов ветки временно пуста
относительно `main`. Файлы на диске не изменились.

## Шаг 3. Пойми, что собираешь

Прежде чем резать на коммиты, посмотри реальное содержимое — группировать нужно по логике
изменений, а не по тому, в каких файлах они лежат:

```bash
git status
git diff --cached --stat
git diff --cached          # вчитайся в сам дифф
```

Затем сними всё со staging, чтобы добавлять обратно порциями:

```bash
git reset            # mixed reset: история та же, но индекс пуст
```

## Шаг 4. Пересобери коммиты по смыслу

Раскладывай изменения на связные группы и коммить каждую отдельно. Один коммит — одна
завершённая мысль (новая фича, один багфикс, один рефакторинг, обновление доков и т.п.).
Не сваливай несвязанное в один коммит и не дроби одну правку на десять.

```bash
git add path/to/related/files        # или git add -p для частичного стейджинга
git commit -m "feat(auth): add JWT login flow"
# повторяй для каждой логической группы
```

Если правки переплелись в одних и тех же файлах, разводи их по коммитам через `git add -p`
и выбирай нужные ханки.

### Формат сообщений: Conventional Commits (английский)

`type(scope): краткое описание в повелительном наклонении`. Scope опционален. Типы:
`feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `style`, `perf`, `build`, `ci`.

**Пример 1**
Изменение: добавили вход по JWT-токену
Коммит: `feat(auth): implement JWT-based login`

**Пример 2**
Изменение: поправили падение при пустом ответе API
Коммит: `fix(api): handle empty response body`

**Пример 3**
Изменение: вынесли парсинг конфига в отдельный модуль, поведение не менялось
Коммит: `refactor(config): extract parser into its own module`

## Шаг 5. Покажи результат и остановись

```bash
git log --oneline main..HEAD
git status
```

Покажи пользователю новую историю и заверши работу. **Не пушь** — это сознательный выбор:
после переписывания истории push потребует force, а это разрушительная операция, которую
человек должен запустить сам. Если ветка уже была запушена, просто предупреди, что для
обновления удалёнки понадобится `git push --force-with-lease`, и оставь решение за ним.

## Если что-то пошло не так

Пока коммиты не удалены сборщиком мусора, всё обратимо. Вернуть ветку к исходному
состоянию:

```bash
git reset --soft $OLD_HEAD     # если переменная ещё в этой сессии
git reflog                     # иначе найди прежний HEAD здесь и сбросься на него
```
