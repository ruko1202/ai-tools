# Do Task

Довести задачу по разработке до рабочего результата: от понимания запроса до готовых, закоммиченных изменений в отдельной ветке. Скилл задаёт дисциплину процесса, чтобы не сломать основную ветку и не утонуть в догадках.

## Язык общения

Общайся с пользователем по-русски на всех этапах — вопросы, отчёты о ходе работы, итоги.

## Рабочий процесс

Иди по шагам по порядку. Не пропускай шаг 1 ради скорости: большая часть плохих правок рождается из непонятого контекста.

### 1. Разберись в задаче

Прежде чем писать код, собери контекст:
- Прочитай релевантные файлы и пойми, как устроен затрагиваемый участок кода — структуру, зависимости, соглашения проекта.
- Сопоставь задачу с тем, что реально есть в репозитории, а не с тем, как «обычно бывает».

Это важно, потому что правка, не вписанная в существующие соглашения, создаёт больше работы, чем экономит.

### 2. Создай отдельную ветку

Всегда автоматически создавай новую ветку под задачу, прежде чем менять код. Имя выбирай сам, осмысленно по сути задачи (например `feat/...`, `fix/...`).

Не спрашивай разрешения и не уточняй имя — просто создай. Это изолирует работу и держит основную ветку чистой.

### 3. Реализуй, действуя на разумных предположениях

Делай задачу. Когда что-то неоднозначно, выбирай наиболее разумный вариант и продолжай — кратко проговаривая принятое допущение, чтобы его можно было поправить.

Задавай уточняющие вопросы ТОЛЬКО когда реально заблокирован: без ответа нельзя двигаться дальше и любой выбор может оказаться грубо неверным (противоречивые требования, отсутствующий доступ, выбор между взаимоисключающими архитектурами с необратимыми последствиями). Во всех остальных случаях не тормози ход вопросами — прогресс важнее уточнений.

### 4. Заверши коммитами

Когда изменения готовы, вызови скилл **do-commits**, чтобы собрать работу в аккуратные, логически сгруппированные коммиты.

Не вызывай `pr` и не пушь автоматически — это отдельные шаги, которые пользователь запускает сам.

## Соседние скиллы

Подключай по необходимости:
- **do-commits** — сгруппировать изменения в чистые коммиты (стандартное завершение задачи).
- **pr** — заголовок и описание для пул-реквеста (по явной просьбе).
- **pretty-commit-from-main** — софт-резет на main и пересборка истории коммитов (по явной просьбе).
